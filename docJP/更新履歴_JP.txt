Rashinban for AI - 更新履歴（日本語）
=======================================

v0.5.0（MCPサーバー）
------
テーマ：複数のAIエージェントが、同じ羅針盤を同時に持てるように。

マルチエージェント対応（MCPサーバーの大規模改修）
- MCP接続方式を stdio から Streamable HTTP（http://127.0.0.1:3766/mcp）
  に変更。サーバーは常駐型となり、複数のAIエージェントが同時に接続
  できるようになった。
  ※従来の stdio 方式では、エージェントごとにサーバープロセスが起動し、
    WebSocketポート（3765）の取り合いになるため実質1エージェント限定
    だった。
- セッション分離：エージェントごとにMCPセッション・ツール状態を分離。
- DOM変化の複製配信：dom_changed は接続中の全セッションのキューに
  複製して配信。get_dom_changes は自分のセッションのキューだけを
  消費する（従来は早い者勝ちで1エージェントが独占していた）。
  キューはセッションごとに上限200件（超過分は古い方から破棄し、
  取得時に droppedCount で通知）。
- 道標のエージェント別保存：michishirube.json を
  { agents: { <agent_id>: 道標 } } のマップ形式に変更。
  save/get/clear_michishirube に agent_id パラメータを追加
  （省略時はセッション固有ID）。旧形式のファイルは初回読み込み時に
  agents.default へ自動移行し、既存データは保持される。
- list_michishirube ツールを新設（agent_id・保存日時・goal・status
  の一覧）。
- get_ui_elements に clientId パラメータを追加。list_connected_tabs
  で取得したclientIdを渡すと特定タブを対象にできる（複数エージェント
  が別々のタブを並行操作するため）。

セキュリティ
- MCP側HTTPエンドポイントも localhost（127.0.0.1）限定バインド。
  加えてDNSリバインディング保護（Hostヘッダ検証）を有効化。
- 拡張機能側の3層セキュリティ（localhost限定・拡張ID検証・トークン
  認証）は無変更。

互換性
- Chrome拡張側は無改修・スキーマ変更なし。
- 【要対応】AIエージェント側のMCP設定は、stdio方式（command: node）
  から HTTP方式（url: http://127.0.0.1:3766/mcp）への変更が必要。
  詳細は取説本編「5. AIエージェントを接続する」を参照。
- stdio 互換は廃止（必要になれば mcp-remote 等のブリッジで代替可能）。


v0.1.1
------
テーマ：より豊かで、より安全なラベル。

追加・改善
- 操作可能要素の「ラベル解決」をアクセシビリティツリー相当に強化。
  `label` フィールドを以下の優先順で解決するようにした：
    1. aria-label
    2. aria-labelledby（参照先要素のテキストを解決）
    3. 関連 <label>（for= / 内包 <label> の両方、el.labels 経由）
    4. ボタンのテキスト / value（<button> と input[submit|button|reset]）
    5. placeholder
    6. title
    7. alt（画像ボタン / 内包画像）
    8. textContent（フォームコントロール以外の要素のみ）
    9. name（最後の手段としての識別子）
- submit / button / reset の input が `value` をラベルとして返すように
  なった（例：<input type="submit" value="送信"> → ラベル「送信」）。
  従来は空だった。
- 表示ラベルが関連 <label> 要素に由来するフォーム項目（ごく一般的な
  パターン）が、空文字ではなくそのラベルテキストを返すようになった。
- ラベル文字列は空白を正規化し、最大80文字で切り詰める。

設計上の線引き（プライバシー）
- 「意味」と「内容」の区別をコードで担保するようにした。要素の
  アフォーダンス（ラベル）は渡すが、テキスト入力に打ち込まれた値
  （ユーザーの内容）は決して返さない。
    * ボタンの value は返す（＝「送信」等のラベルだから）。
    * テキスト入力の value は返さない（＝ユーザー入力の内容だから）。
    * <select> の選択肢テキストはラベルに使わない（内容漏れになるため）。
  これにより「ページ内容は渡さない、どこを操作するかだけ渡す」という
  本ツールの思想を保ったまま、「どれが送信ボタンか分からない」問題を
  解消している。

互換性
- スキーマ変更なし。出力オブジェクトの形は同じで、`label` フィールドの
  埋まり方がより完全になっただけ。MCPサーバーおよび道標（michishirube）
  連携は無改修。既存のAI側パースもそのまま動作する。

今回は見送った項目（設計判断による）
- 道標 ENV ブロックへの「今フォームのどのステップにいるか」自動判定。
  （任意のステップUIに対して判定が不確実であり、「未観測の項目を推測で
  埋めない」という道標の原則と衝突するため。）
- 独立した `name` セマンティックフィールドの新設。
  （スキーマ安定性を優先し、当面は label のフォールバックに留める。）


v0.1.0
------
- 初回リリース。
- Phase 1：操作可能要素の抽出（CSSセレクタ・座標・状態）と種別ヒント
  判定（日付ピッカー検知等）。
- Phase 2：WebSocket 経由のローカル MCP サーバー。再接続・指数
  バックオフ付き。
- Phase 3：MutationObserver による動的要素の検知・通知（debounce付き）。
- Phase 4：3層セキュリティ（localhost 限定バインド、Chrome 拡張 ID
  検証、トークン認証）。
