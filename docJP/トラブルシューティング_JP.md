# トラブルシューティング

## 症状：MCPサーバーが起動しているか分からない

**確認方法**

```powershell
Get-Process node -ErrorAction SilentlyContinue
```
→ 何も表示されなければ、サーバーは起動していません。

```powershell
Get-NetTCPConnection -LocalPort 3765, 3766 -ErrorAction SilentlyContinue
```
→ 何も表示されなければ、サーバーは起動していません（3765＝拡張機能側、3766＝AIエージェント側）。

**対処**

```powershell
cd "<プロジェクトのパス>\mcp-server"
node server.js
```
以下の2行が表示されれば起動成功です。

```
[Rashinban] WebSocketサーバー起動: ws://127.0.0.1:3765
[Rashinban] MCPサーバー起動: http://127.0.0.1:3766/mcp
```

v0.5.0以降、サーバーは常駐型です。AIエージェントから使う間は起動したままにしてください。

---

## 症状：「ポート3765（または3766）が使用中です」と表示されて起動できない

**原因**
既に別のサーバープロセスが同じポートを使用しています。よくあるのは以下の2つです。

1. 前回起動したサーバーを閉じ忘れている
2. **旧バージョン（v0.4.x以前）のstdio方式の設定が、AIエージェント側に残っている**：旧設定のクライアント（LM Studio、Claude Desktop等）が `server.js` を裏で起動し、ポートを掴んでいるケース。この場合はプロセスを止めるだけでなく、そのクライアントのMCP設定をHTTP方式（取説の手順5）に変更してください。変更しない限り、クライアントの再起動のたびに同じ競合が起きます。

**対処**

1. ポートを使っているプロセスIDを調べる
   ```powershell
   Get-NetTCPConnection -LocalPort 3765, 3766 | Select-Object LocalPort, OwningProcess
   ```
2. 表示されたIDのプロセスを停止する
   ```powershell
   Stop-Process -Id <表示されたID> -Force
   ```
3. 停止を確認してから再起動
   ```powershell
   Get-Process node -ErrorAction SilentlyContinue
   node server.js
   ```

## 症状：認証トークンを誤って画面に表示してしまった／第三者に見られた可能性がある

**対処（トークンの差し替え）**

1. サーバーを停止する（起動しているターミナルで `Ctrl + C`）
2. `config.json` を削除する
   ```powershell
   Remove-Item "<プロジェクトのパス>\mcp-server\config.json" -ErrorAction SilentlyContinue
   ```
3. サーバーを再起動する（新しいトークンが自動生成されます）
   ```powershell
   node server.js
   ```
4. 表示された新しいトークンを、拡張機能のオプション画面に再設定する
   - `chrome://extensions` → 拡張機能の「詳細」→「拡張機能のオプション」
   - トークン欄に貼り付けて保存

※ `config.json` には「承認済み拡張機能ID」も記録されているため、削除すると拡張ID側の自動承認も初回接続からやり直しになります（次項を参照）。

## 症状：`トークン不一致で接続拒否: extensionId=undefined`

**原因**
主に2パターンあります。

1. **config.json を削除・再生成した直後**：拡張機能側がまだ古いトークン・古い接続情報を持ったまま再接続を試みている
2. **複数タブを開いたまま拡張機能を更新した**：更新前から開いていたタブが、リロードされないまま古い接続情報で再接続を試み続けている

**対処**

1. `chrome://extensions` で拡張機能自体をリロード（↺ボタン）
2. 拡張機能が動作している**全てのタブ**をリロード（F5）— 1タブでも更新し忘れると、そのタブだけエラーを出し続けます
3. オプション画面で新しいトークンが正しく保存されているか確認

**正常化の確認**
ターミナルに以下のようなログが続けば回復しています。
```
[Rashinban] 認証成功: <拡張ID> → <URL>
```
リロードし忘れたタブが残っている場合、`トークン不一致`のログと`切断`のログが数回出てから自然に止まります（古い接続が破棄されるため）。それ以上続く場合は、未リロードのタブが残っていないか再確認してください。

---

## 補足：トークンは起動のたびに変わるのか？

変わりません。`config.json` が存在する限り同じトークンが再利用されます。`config.json` を削除した状態で再起動すると新しいトークンが生成されます。誤って画面に表示してしまった場合のみ、上記の手順で差し替えてください。

---

## 症状：AIエージェントからMCPサーバーに接続できない（ツール一覧に出ない）

**原因と対処（上から順に確認）**

1. **サーバーが起動していない**：v0.5.0以降は常駐型です。AIエージェントがサーバーを起動してくれる方式ではなくなったため、`node server.js` を自分で起動しておく必要があります（本ページ先頭の確認方法を参照）。
2. **接続設定が旧stdio方式のまま**：`command: node` で `server.js` のパスを指定する旧形式は使えません。接続URL `http://127.0.0.1:3766/mcp` を指定するHTTP方式に変更してください（取説の手順5）。
3. **URLの間違い**：パスは `/mcp` まで含めて指定します（`http://127.0.0.1:3766` だけでは接続できません）。

**正常化の確認**
サーバーのターミナルに `[Rashinban] MCPセッション開始: <セッションID>` が出れば、AIエージェント側との接続は成立しています。

---

## 補足：複数のAIエージェントから同時に使える？

使えます（v0.5.0以降）。それぞれのエージェントが同じURL（`http://127.0.0.1:3766/mcp`）に接続するだけで、セッションは自動的に分離されます。DOM変化の通知は全エージェントに配信され、道標は `agent_id` で分けて保存できます。詳細は取説の「マルチエージェント同時利用」を参照してください。

## 症状：`Uncaught Error: Extension context invalidated.`（タブをリロードしても消えない）

**原因**
拡張機能をリロード（`chrome://extensions`の↺）した際、既に開いていたタブのコンテンツスクリプトが孤立し、F5（タブのリロード）だけでは復旧しないことがあります。コンソールに繰り返し `Extension context invalidated` が出続ける場合、これに該当します。

**対処**

タブのF5リロードで直らない場合は、拡張機能自体を一度無効化→有効化してください。

1. `chrome://extensions` を開く
2. Rashinban for AI のトグルスイッチをOFFにする
3. 数秒待ってから、再度ONにする
4. 影響を受けていたタブをすべてF5でリロードする

これで直らない場合は、`chrome://extensions` でいったん拡張機能を削除し、改めて「パッケージ化されていない拡張機能を読み込む」から読み込み直してください（設定したトークンは `chrome.storage.local` に保存されているため、削除すると失われます。再設定が必要です）。

**正常化の確認**
MCPサーバー側のターミナルに、以下のようなログが複数のタブに対して出れば回復しています。

```
[Rashinban] 認証成功: <拡張ID> → <URL>
```

## 症状：AIに「ツールが使えない／その機能はありません」と言われる

**原因**
よくあることです。拡張機能はすでに接続済みで、ツールも使える状態のはずです。多くの場合、ツールを登録する前から続けているチャットで、モデルが「ツールが無かった時点」の状態を覚えたままになっています。

**対処**

新しいチャットを始めてから、もう一度試してください。

---

## 症状：ツール呼び出しは成功しているのに、AIの返事が妙に抽象的（具体的な要素名や座標を答えない）

**原因**
これはエラーが出ないタイプの不具合です。MCPサーバー側のターミナルログには取得した要素数（例：「38個の操作可能要素を検出」）が正しく出ているのに、AIの返事は「ボタンやリンクが含まれています」のような一般論で終わる場合、これに該当します。

ローカルLLM（ollama等）のコンテキストウィンドウ（`num_ctx`）が小さすぎて、取得したJSONデータ（要素数が多いほど大きくなる）が途中で切られ、AIに実データが渡っていないことが原因です。エラーにならないため気づきにくく、AI自身も「データが切られている」とは言わずに、それらしい一般論で答えてしまうことがあります。

**対処**

1. 使用しているクライアント（AnythingLLM等）のモデル設定で、Context Window（コンテキストウィンドウ／Max Tokens）の値を確認する
2. 自動設定や2048〜4096のような小さい値になっている場合は、**8192以上（推奨）** に増やして保存する
3. もう一度ツールを呼び出し、AIの返事に具体的な要素名・座標が含まれるか確認する

**正常化の確認**
「取得した要素を3つ具体的に教えて」のように聞いて、実際のラベルや種類（リンク／ボタンなど）が返ってくれば回復しています。

## 症状：Claude Codeから使うと、MCPサーバー（localhost）に接続できない

**原因**
Claude Codeには任意のサンドボックス機能（`/sandbox`）があり、これを有効にしている場合、localhost（127.0.0.1 / ::1）宛の接続がブロックされる既知の不具合が報告されています。`sandbox.network.allowedDomains`にlocalhostや127.0.0.1を追加しても、Seatbeltレベルでの遮断が先に働き、塞がれてしまうケースがあります（参考：GitHub `anthropics/claude-code` Issue #28018、2026年6月時点で未解決）。

**対処**

1. `/sandbox`を有効にせずに本ツールを使う（デフォルトのClaude Codeはユーザー自身のネットワークで動くため、この制限は受けません）
2. `/sandbox`を使いたい場合は、上記issueの状況を確認しつつ、localhost宛の接続が実際に通るか別途検証してください。現時点では本ツール側で確実に回避する方法はありません。

**正常化の確認**
MCPサーバーのターミナルに `[Rashinban] 認証成功: <拡張ID> → <URL>` のログが出れば、接続は問題なく通っています。

※ これはClaude Code側の既知issueであり、本ツールの不具合ではありません。

## 症状：Cowork（Claude desktopのCoworkモード）からは、ブラウザでの動作確認自体ができない

**原因**
CoworkモードはクラウドのサンドボックスでClaudeを動かしており、ネットワークアクセスは管理者が設定したホスト側の許可リストに固定されています。本ツールのMCPサーバー（localhost）はもちろん、許可されていない一般サイト（実例：peraichi.com、ja.wikipedia.org）も「組織のポリシーでブロック」されて開けません。

これはClaude Codeの`/sandbox`issue（上記）とは別物で、Cowork固有のホスト側制限です。Coworkに搭載されているClaude in Chrome拡張（ブラウザ操作ツール）を使っても、リクエストが許可リストでブロックされるため、ページ内容の取得や要素検出はできませんでした。

**対処**

1. 本ツールの動作確認はClaude Code（デフォルト設定、`/sandbox`なし）やローカルのLM Studio等から行う
2. Cowork側でどうしても試したい場合は、管理者にAdmin設定 → Capabilities でネットワークアクセス設定を確認してもらう（個人利用のCoworkでは設定できない場合があります）

**正常化の確認**
この制限はCowork環境固有のため、本ツール側で直すことはできません。Claude Codeなど別環境に切り替えて、上の項目の確認方法で動作確認してください。

## 備考

- LM Studio＋自作MCPサーバーの既存構成と同じ経路で組み込めます（追加コストなし）
- SPA（シングルページアプリ）はURLが変わらずに画面が切り替わることがあるため、ページ遷移後に要素が取得できない場合はこの可能性を確認してください
- reCAPTCHAなどのボット対策は突破できません（設計上、対象外です）
